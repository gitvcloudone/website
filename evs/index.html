What is Amazon EVS?

VMware Cloud Foundation,
natively on AWS

Amazon Elastic VMware Service (Amazon EVS) is a native AWS service that lets you deploy and run VMware Cloud Foundation (VCF) directly within your own Amazon VPC. It runs on dedicated EC2 bare-metal instances — no shared infrastructure, no hypervisor between you and the metal.

EVS was jointly engineered by AWS and Broadcom. AWS operates and supports the infrastructure (L1/L2), while Broadcom provides L3 support for VCF. You retain full root access to vSphere, NSX Manager, vSAN, and SDDC Manager.

Bring your own VCF subscription (BYOS) from Broadcom. AWS bills separately for EC2 instances, Route Server endpoints, and the EVS control plane.

Network Architecture

EVS uses dedicated VLANs inside the VPC for all VMware traffic. The VPC Route Server dynamically propagates NSX overlay routes to the VPC route table via BGP, eliminating static route management.

EVS provisions all VLANs automatically. The Route Server exchanges BGP routes between NSX Tier-0 Edge nodes and the VPC, enabling dynamic routing without static route management.

Use Cases

From lift-and-shift to cloud-native

EVS supports every stage of the cloud journey — from DC extension to modern application platforms.

Cloud Extension & Data Center Expansion

Extend your on-premises data center to AWS without re-architecting workloads or retraining staff.

  • Burst capacity for seasonal demand spikes
  • Data center consolidation / footprint reduction
  • HCX live migration — no downtime, no IP changes
  • Retain on-prem runbooks and tooling

Disaster Recovery

Use EVS as a cost-effective secondary site with fast RTO/RPO using VMware Site Recovery Manager and vSphere Replication.

  • Temporary DR capacity during on-prem maintenance
  • Site Recovery Manager (SRM / VLSR) supported
  • HCX L2 extension for seamless network stretch
  • Advanced Cyber Recovery (ACC) add-on available

Rapid Cloud Migration

Migrate VMware workloads to AWS in days, not months — without changing IP addresses, rewriting applications, or retraining teams.

  • True lift-and-shift via HCX bulk migration
  • Legacy system retirement at cloud pace
  • Gradual AWS service adoption post-migration
  • Same vSphere/NSX skills, zero retraining

VCF as a Service

Shift from CapEx to OpEx with on-demand VCF consumption. AWS bills only what you use — no hardware procurement cycle.

  • Board mandate: CapEx → OpEx
  • On-demand, 1-year, or 3-year subscription terms
  • Self-managed or AWS Partner managed operations
  • Short-term dev/test, training, hackathons

AI / Analytics Workloads

Keep data gravity on EVS while accessing native AWS AI/ML services — no data movement required.

  • Direct access to SageMaker, Bedrock, Rekognition
  • Analytics without refactoring existing VMs
  • Private AI Foundation (PAIF) add-on on roadmap
  • Proof-of-value pilots before re-architecture

Modern App Platform

Run Kubernetes natively alongside VMs using VMware Kubernetes Service (VKS) and Tanzu on EVS.

  • VKS (VMware Kubernetes Service) available
  • Data Services Manager (DSM) for databases
  • VCF Automation for self-service app catalog
  • NSX micro-segmentation for app security
Shared Responsibility

Who manages what

EVS follows the AWS Shared Responsibility Model. AWS owns the physical infrastructure; you own the VCF stack and everything above it.

AWS Manages

  • Physical bare-metal hardware (EC2 i4i.metal)
  • AWS global network infrastructure
  • VPC, Route Server, Route 53, IAM services
  • L1 / L2 support for EVS and EC2
  • Physical security of AWS data centers
  • EVS control plane (CreateEnvironment API)
  • Hardware replacement and failure remediation

Customer / Partner Manages

  • VCF lifecycle (patches, upgrades via SDDC Manager)
  • vSphere VM and cluster configuration
  • NSX network policy and segmentation
  • Guest OS and application security
  • VCF licensing (BYOS from Broadcom)
  • IAM roles and access control
  • Regulatory and compliance requirements
  • Backup, DR configuration, data protection
