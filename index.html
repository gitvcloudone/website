Mastering Amazon EVS: Prerequisites and Automation

Posted on March 25, 2026

Amazon Elastic VMware Service (EVS) is a game-changer for enterprises looking to lift-and-shift their VMware workloads to AWS. Unlike traditional managed services, EVS gives you full administrative control over the VMware stack while leveraging the scale and elasticity of the AWS Cloud. However, success starts with a rock-solid foundation.

Understanding the Prerequisites

Before you deploy your first EVS environment, you must prepare your AWS infrastructure. The most critical component is the VPC Underlay.

  • Dedicated VPC: A large CIDR (e.g., /16) is recommended to accommodate both AWS native services and VMware management VLANs.
  • Service Access Subnet: A subnet (minimum /28) in your target AZ to host the VPC Route Server endpoints.
  • Reserved VLAN CIDRs: You must identify four non-overlapping CIDR blocks (/28 to /24) for Host Management, vMotion, vSAN, and NSX Uplinks.
  • DNS & NTP: A custom DHCP Option Set pointing to Route 53 Inbound Resolvers and the Amazon Time Sync Service.

Automation with Terraform

Manually configuring these networking components is prone to error. We recommend using Terraform to ensure a repeatable and consistent deployment environment.

1. The VPC Infrastructure

This template creates the base VPC and the necessary Service Access Subnet required for the EVS Route Server.

resource "aws_vpc" "evs_vpc" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = {
    Name = "evs-underlay-vpc"
  }
}

resource "aws_subnet" "service_access" {
  vpc_id            = aws_vpc.evs_vpc.id
  cidr_block        = "10.0.0.0/28"
  availability_zone = "us-west-2a"

  tags = {
    Name = "evs-service-access-subnet"
  }
}

2. VPC Route Server & Propagation

The Route Server is the "bridge" between the VMware NSX overlay and your AWS VPC routing table.

# Example of enabling route propagation for EVS
resource "aws_vpn_gateway" "evs_gateway" {
  vpc_id = aws_vpc.evs_vpc.id
}

resource "aws_vpn_gateway_route_propagation" "evs_propagation" {
  vpn_gateway_id = aws_vpn_gateway.evs_gateway.id
  route_table_id = aws_vpc.evs_vpc.main_route_table_id
}

Conclusion

Amazon EVS offers unparalleled flexibility for VMware administrators, but it requires a disciplined approach to networking. By automating the underlay with Terraform, you reduce the risk of configuration drift and accelerate your journey to a hybrid cloud environment.
