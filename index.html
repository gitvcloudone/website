1 Phase 1: Bare-Metal Readiness

1.1 Validating Instance Quotas

Amazon EVS utilizes the i4i.metal instance type. A minimum of 4 nodes is required for the initial VCF management domain cluster. Ensure your AWS Service Quotas for Running Dedicated i4i.metal Hosts is set to at least 4 per region before proceeding.

2 Phase 2: Multi-VLAN Networking (Underlay)

Amazon EVS automatically provisions 10 VLANs during environment bring-up. All VLAN subnets must fall within your VPC CIDR (minimum /28, maximum /24) and must be non-overlapping. The one exception is the HCX Uplink VLAN, which must be exactly /28 if HCX is used.

Deployment Note: All 10 VLANs above are provisioned automatically by Amazon EVS during bring-up — do not pre-create them. The one subnet you must create manually beforehand is the Service Access Subnet (used for the Route Server and DNS Resolver endpoints). This subnet is not a VLAN managed by EVS.

4 Phase 4: VPC Route Server & BGP

The VPC Route Server is the BGP bridge between NSX-T and the AWS underlay. It dynamically propagates NSX overlay routes into your VPC route tables, enabling AWS-native services to reach VMware workload segments without static routes. This phase must be completed before submitting the EVS environment creation request.

4.1 Create the VPC Route Server

In the AWS Console, navigate to VPC › Virtual Private Cloud › Route Servers and click Create route server. Configure the following:

FieldValueNotes
Nameevs-route-server-01Any descriptive label
Amazon Side ASN64512–65534 (e.g. 65100)Must be a different value than the NSX Edge ASN (default 65000)
Persist RoutesEnabledPreserves routes for 1–5 min after a BGP session drops; prevents route flaps
SNS NotificationsOptionalRecommended for production — alerts on BGP state changes

Wait for the Route Server status to reach Available before proceeding.

4.2 Associate the Route Server with the Service Access Subnet

Select the Route Server › Associations tab › Associate route server. Select your Service Access Subnet (the one pre-created in Phase 2). AWS will automatically deploy two Route Server endpoints into that subnet — one per availability zone. Both endpoints must reach Available state before you create peers.

Important: Each endpoint receives a private IP from the Service Access Subnet. Note these IPs — you will need them when configuring BGP neighbors on the NSX Tier-0 gateway in step 4.5.

4.3 Create Route Server Peers (One per NSX Edge)

Navigate to Route server peers tab › Create route server peer. Create one peer per NSX Edge node (two total). The peer address is the IP that will be assigned to each NSX Edge's uplink interface in the NSX Uplink VLAN — not the management IP.

FieldPeer 1 (Edge 01)Peer 2 (Edge 02)
Route Server EndpointEndpoint 1 IDEndpoint 2 ID
Peer Address10.0.16.10 (NSX Uplink VLAN)10.0.16.11 (NSX Uplink VLAN)
Peer ASN6500065000
Liveness DetectionBGP KeepaliveBGP Keepalive
Critical: Always use BGP Keepalive as the liveness detection method. BFD (Bidirectional Forwarding Detection) is not supported for Amazon EVS and will cause undefined session behavior.

4.4 Enable Route Propagation on the VPC Route Table

Navigate to the Route Server › Propagations tab › Create propagation. Select the route table(s) that need to receive NSX overlay routes — at minimum, the Service Access Subnet route table and your workload subnet route tables.

Common failure point: Every route table configured for propagation must have at least one explicit subnet association. Route tables with only an implicit main-route-table association will silently fail to receive BGP routes. Verify this with aws ec2 describe-route-tables before proceeding.

4.5 Configure NSX Tier-0 BGP (VMware Side)

After EVS bring-up completes, log into NSX Manager and navigate to Networking › Tier-0 Gateways. Edit the Tier-0 gateway and configure BGP under the Routing section:

BGP SettingValue
Local AS Number65000 (must match Peer ASN set on Route Server)
BGP StatusEnabled
Graceful RestartEnabled
Keepalive Timer5 seconds
Hold Time60 seconds (minimum 3× keepalive)

Then add two BGP Neighbors under BGP › Neighbors — one for each Route Server endpoint IP noted in step 4.2:

Neighbor FieldNeighbor 1Neighbor 2
Neighbor AddressRoute Server Endpoint 1 IPRoute Server Endpoint 2 IP
Remote AS65100 (Route Server ASN)65100 (Route Server ASN)
Source Address10.0.16.10 (Edge 01 uplink)10.0.16.11 (Edge 02 uplink)
Route Filterrfc-1918-allowrfc-1918-allow
Max Hop Limit22

Finally, enable Route Redistribution on the Tier-0 to advertise Connected and NSX Static routes. This causes NSX to push overlay VLAN prefixes to the Route Server.

4.7 Networking Prerequisites & Port Requirements

BGP runs on TCP port 179. Ensure the following is in place before expecting sessions to establish:

RequirementDetail
Security GroupsAllow TCP 179 inbound/outbound between NSX Uplink VLAN CIDR and Service Access Subnet CIDR
Network ACLsAllow TCP 179 and ephemeral ports 49152–65535 bidirectionally (NACLs are stateless)
Route Server ASN ≠ Peer ASNUsing the same ASN on both sides (e.g. both 65000) will prevent BGP negotiation entirely
Unique peer IPsEach Route Server peer must use a unique IP from the NSX Uplink VLAN; duplicates cause only one session to be active
Explicit subnet associationAll propagated route tables must have at least one explicit subnet association (not just the implicit main association)

⚡ Automation: Terraform Deployment

The following modules are included in the EVS-PreREQ-Terraform kit and automate the complete underlay setup.

main.tf — VPC, DNS Resolver, Route Server v1.5 
# 1. Create Base VPC Underlay
resource "aws_vpc" "evs_vpc" {
  cidr_block           = var.vpc_cidr
  enable_dns_hostnames = true
  enable_dns_support   = true
  tags = { Name = "evs-vpc" }
}

# 2. Inbound DNS Resolver for SDDC Components
resource "aws_route53_resolver_endpoint" "evs_dns" {
  name               = "evs-dns-resolver"
  direction          = "INBOUND"
  security_group_ids = [aws_security_group.dns_sg.id]

  ip_address { subnet_id = aws_subnet.service_access.id; ip = "10.0.1.100" }
  ip_address { subnet_id = aws_subnet.service_access.id; ip = "10.0.1.110" }
}

# 3. VPC Route Server for NSX Edge BGP Peering
resource "aws_vpc_route_server" "evs_rs" {
  amazon_side_asn = 64512
}

resource "aws_vpc_route_server_peer" "nsx_edge_01" {
  route_server_id = aws_vpc_route_server.evs_rs.route_server_id
  peer_address    = "10.0.10.251"
  bgp_options { peer_asn = 65000 }
}

resource "aws_vpc_route_server_peer" "nsx_edge_02" {
  route_server_id = aws_vpc_route_server.evs_rs.route_server_id
  peer_address    = "10.0.10.252"
  bgp_options { peer_asn = 65000 }
}
dns.tf — Route 53 Private Zones & Records v1.5 
# Forward Zone
resource "aws_route53_zone" "forward" {
  name = var.domain_name
  vpc { vpc_id = aws_vpc.evs_vpc.id }
}

# A Records — dynamically built from vcf_components variable
resource "aws_route53_record" "vcf_a" {
  for_each = var.vcf_components
  zone_id  = aws_route53_zone.forward.zone_id
  name     = "${each.key}.${var.domain_name}"
  type     = "A"
  ttl      = 300
  records  = [each.value]
}

# PTR Records — extracts last octet from IP automatically
resource "aws_route53_record" "vcf_ptr" {
  for_each = var.vcf_components
  zone_id  = aws_route53_zone.mgmt_reverse.zone_id
  name     = "${element(split(".", each.value), 3)}.10.0.10.in-addr.arpa."
  type     = "PTR"
  ttl      = 300
  records  = ["${each.key}.${var.domain_name}."]
}
