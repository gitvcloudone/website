Architecting Enterprise SDDC: Amazon EVS Professional Guide

Mastering Amazon EVS: Comprehensive Prerequisite Guide

March 25, 2026 | Deployment & Hybrid Cloud Architecture

Amazon Elastic VMware Service (EVS) enables organizations to run VMware Cloud Foundation (VCF) directly on AWS bare-metal infrastructure. A successful Software-Defined Data Center (SDDC) deployment depends on a precisely configured AWS "underlay" networking and identity foundation.

This guide details the mandatory manual configuration steps required before initiating an SDDC bring-up, including instance validation, networking, DNS, and IP pool planning.

1 Phase 1: Capacity & IAM Readiness

1.1 Validating Instance Quotas

Path: Console > Service Quotas > AWS Services > Amazon EC2

Search for **"Running Dedicated i3.metal Hosts"**. Verify that the "Applied quota value" is at least **3** (the minimum for a production management cluster). For modern high-performance workloads, ensure you also check quotas for **i3en.metal** or **i4i.metal** if supported in your target region.

1.2 Provisioning the Service-Linked Role (SLR)

Path: Console > IAM > Roles > Create Role

AWS requires the AWSServiceRoleForEVS to manage VPC resources (ENIs, Route Tables) on your behalf. Select **"AWS Service"**, search for **EVS**, and complete the role creation process.

2 Phase 2: VPC Underlay & IP Pool Planning

The VPC acts as the transport layer for the SDDC. Beyond the base VPC, you must reserve specific CIDR ranges for the internal VMware VLANs.

2.1 Create the VPC and Service Access Subnet

Path: Console > VPC > Your VPCs > Create VPC

ComponentRecommended Value
VPC CIDR10.0.0.0/16
Service Access Subnet10.0.1.0/24 (Availability Zone: us-west-2a)

2.2 Reserved IP Pools for VLANs

During SDDC bring-up, you must provide non-overlapping CIDR blocks for the following VMware-managed networks. These should **not** be created as VPC subnets, but rather reserved for EVS use:

3 Phase 3: Route 53 DNS Infrastructure

VMware Cloud Foundation is highly dependent on both Forward and Reverse DNS resolution. Failure to provide PTR records will result in an SDDC Manager bring-up failure.

3.1 Mandatory DNS Records

Create a **Private Hosted Zone** (e.g., vcloudone.internal) and associate it with your EVS VPC. The following records are mandatory:

ComponentForward Record (A)Reverse Record (PTR)
SDDC Managersddc-manager.vcloudone.internal10.0.10.10
vCenter Servervcenter.vcloudone.internal10.0.10.11
NSX Manager 01nsx-mgr-01.vcloudone.internal10.0.10.12
ESXi Host 01esxi-01.vcloudone.internal10.0.11.21

3.2 Inbound Resolver Endpoints

Provision **Route 53 Inbound Resolver Endpoints** in your Service Access Subnet. This allows the VMware management appliances to resolve records within the AWS private hosted zones.

4 Phase 4: VPC Route Server & BGP Peering

4.1 Route Server Configuration

Path: Console > VPC > Route Servers > Create Route Server

Set the **Amazon-side ASN to 64512**. Once the Route Server is active, create two **BGP Peers** to connect to your VMware NSX Edges:

  • Peer 1 IP: 10.0.10.251 (ASN: 65000)
  • Peer 2 IP: 10.0.10.252 (ASN: 65000)

Crucial: Enable **Route Propagation** on the VPC Route Table to allow VMware overlay routes to populate automatically.

Automation: Professional Terraform Template

To ensure a consistent, repeatable deployment that avoids manual configuration drift, use the following Terraform template.

main.tf (Production-Grade) v1.3 
# VPC & Core Networking
resource "aws_vpc" "evs_vpc" {
  cidr_block = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support = true
  tags = { Name = "evs-underlay" }
}

# Route 53 Resolver for VMware Integration
resource "aws_route53_resolver_endpoint" "evs_dns" {
  name = "evs-inbound-resolver"
  direction = "INBOUND"
  security_group_ids = [aws_security_group.dns_sg.id]

  ip_address { subnet_id = aws_subnet.service_access.id; ip = "10.0.1.100" }
  ip_address { subnet_id = aws_subnet.service_access.id; ip = "10.0.1.110" }
}

# VPC Route Server for BGP Propagation
resource "aws_vpc_route_server" "evs_rs" {
  amazon_side_asn = 64512
  tags = { Name = "evs-route-server" }
}

resource "aws_vpc_route_server_peer" "nsx_edge_01" {
  route_server_id = aws_vpc_route_server.evs_rs.route_server_id
  peer_address = "10.0.10.251"
  bgp_options { peer_asn = 65000 }
}
