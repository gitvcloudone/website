Architecting the Hybrid Cloud: VMware & AWS Integration

Amazon EVS: The Definitive Prerequisite Guide

March 25, 2026 | By David | Deployment & Automation

Deploying **Amazon Elastic VMware Service (EVS)** is a significant milestone for any hybrid cloud strategy. It brings the power of VMware Cloud Foundation (VCF) directly onto AWS bare-metal infrastructure. However, the success of your SDDC hinges on the quality of your **AWS Underlay**.

In this guide, we provide the exact, step-by-step manual process for preparing your environment, followed by a production-ready Terraform template to automate the entire stack.

Phase 1: Capacity & IAM Readiness

Before touching any networking, ensure your AWS account is authorized to run the EVS service.

1. Check Service Quotas

Navigate to Service Quotas > AWS Services > Amazon EC2. Search for Running Dedicated i3.metal Hosts or i4i.metal. You must have a limit of at least 3 per region. If it is lower, request an increase immediately.

2. Enable the Service-Linked Role

AWS requires a specific role to manage EVS resources on your behalf. While often created automatically, it is best practice to provision the AWSServiceRoleForEVS manually via IAM if it doesn't exist.

Phase 2: Network Foundation (VPC Underlay)

This is where the ESXi hosts and management appliances will live.

  1. Create a dedicated VPC: Use a CIDR block that offers long-term growth (e.g., 10.0.0.0/16). Ensure DNS Hostnames and DNS Support are enabled.
  2. Provision a Service Access Subnet: This subnet hosts your Route Server endpoints and DNS Resolvers. A /24 or /28 is sufficient.
  3. Configure DHCP Option Sets: You must create a custom set that points to your VPC DNS resolvers and the Amazon Time Sync Service (169.254.169.123).
Crucial Step: Reserve your VLAN CIDRs now! You will need non-overlapping ranges for Host Management, vMotion, vSAN, and NSX Uplinks during the final deployment phase.

Phase 3: DNS & Routing Logic

VMware is a DNS-heavy architecture. Without proper Forward and Reverse lookups, the SDDC Manager will fail during bring-up.

1. Route 53 Resolver Endpoints

Create an Inbound Resolver Endpoint within your Service Access Subnet. This allows your VMware management appliances to resolve AWS names and vice versa.

2. VPC Route Server (BGP)

Provision the Amazon VPC Route Server. This is mandatory for EVS to communicate with the VMware NSX edges. You must enable Route Propagation on your VPC Route Tables so that VMware workload routes are automatically injected into AWS.

Full Automation: The "Master" Terraform Template

Don't want to do it manually? Below is the Production-Ready Terraform Template that automates every step mentioned above: VPC, Subnets, DNS Zones, A/PTR records, and the VPC Route Server with BGP peering.

main.tf Production Grade 
# 1. Create VPC Underlay
resource "aws_vpc" "evs_vpc" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true
  tags = { Name = "evs-production-vpc" }
}

# 2. Service Access Subnet
resource "aws_subnet" "service_access" {
  vpc_id            = aws_vpc.evs_vpc.id
  cidr_block        = "10.0.1.0/24"
  availability_zone = "us-west-2a"
}

# 3. VPC Route Server (BGP Connection)
resource "aws_vpc_route_server" "evs_rs" {
  amazon_side_asn = 64512
}

resource "aws_vpc_route_server_vpc_association" "evs_rs_vpc" {
  route_server_id = aws_vpc_route_server.evs_rs.route_server_id
  vpc_id          = aws_vpc.evs_vpc.id
}

# 4. Route 53 Forward & Reverse Zones
resource "aws_route53_zone" "forward" {
  name = "vcloudone.internal"
  vpc { vpc_id = aws_vpc.evs_vpc.id }
}

resource "aws_route53_zone" "mgmt_reverse" {
  name = "10.0.10.in-addr.arpa"
  vpc { vpc_id = aws_vpc.evs_vpc.id }
}

# 5. Service-Linked Role
resource "aws_iam_service_linked_role" "evs" {
  aws_service_name = "evs.amazonaws.com"
}
